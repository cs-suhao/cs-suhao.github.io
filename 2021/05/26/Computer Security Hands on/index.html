Section 2 Set-UID Program and Vulnerabilities

The need for Privileged Program

因为粗粒度的ACL无法满足要求，例如我们需要修改密码，就需要修改/etc/shadow，但是这个文件只能被root修改，当用户需要修改自己的密码时，并没有权限修改自己的密码。一般有两种方式实现，第一种是系统运行一种背景程序privileged process，需要操作时，用户向背景程序发送请求，真正的命令由背景程序运行。第二种就是使用set-uid。

How Set-UID Program Work

当运行一个program，会生成一个process，在process中有uid。之后该进程进入file时，os会检查进程的uid，再决定该进程有无操作file的权限。细致来说，一个进程有三个不同的uid：

  • real uid-真实用户ID
  • effective uid-有效用户ID
  • 保留用户ID

set-UID 设置的是effective uid，更改的是该程序的所有者。例如，当程序被用户ID为5000的用户执行，进程的真实用户ID和有效用户ID都是5000，而如果执行了set-UID以后，该程序的真实用户ID还是5000，而有效用户ID则属于程序的所有者，若所有者为root，则有效用户ID是0。

What Can Go Wrong in Privileged Programs?

举一个超人的例子，如果穿超人制服的普通人不想按照设定的程序执行，该如何做？下面的例子就是这样，按照设定，普通人应该向前飞，再向右飞最终解救人质。但是超人的飞行方向是面朝下，如果普通人改成面朝上飞，那么他就可以去左边的银行抢劫。如果超人修改了方向，不再使用相对方位，而使用绝对方位，即向东飞向西飞，该如何做到攻击？攻击者可以修改坐标，即修改东和西的计算结果，因为向东向西是根据地理位置算出来的，攻击者直接把标准修改来欺骗程序。

Attacks via Environment Variables

在set-UID程序中，恶意用户可以修改环境变量来进行漏洞攻击。在进程中，会有环境变量的键值对-<name=value>集合，键值对集合是由父进程继承而来的，该进程的子进程也会相应继承其父进程的环境变量。这些环境变量如果没有用到，则对进程没有影响，如果用到了则会由影响。在set-UID程序中，它的父进程是shell进程， 此时shell进程中的euid=seed，而调用了set-UID进程之后，euid=root，并且一些环境变量会被继承。

例如下面的程序，如何利用set-UID实现攻击。因为system函数首先会调用/bin/sh程序，接着找到cal，但是寻找的路径是相对路径，一般会在/usr/bin/cal中找，但是如果我们指定路径，那它就可以运行我们写的任何名为cal的程序。

另一种方法，恶意用户可以通过将自己的程序混入某一个程序的执行程序库中，也可以进行恶意操作。这就是Dynamic-Link Library。程序的进程内存中，分成静态链接和动态链接两种，静态链接需要将程序拷贝至自己的二进制编译文件中，这就导致程序复用的时候会浪费内存。而动态链接则在编译的时候，通过链接直接指向所包含的程序，而不用拷贝。

Attacks via Explicit User Inputs

前面的变量都是通过该边环境变量这样的隐形变量，它一般不会显示，除非知道的人才能够修改。而这一小节，我们可以用明确的用户变量来实现攻击，可以看到下面的程序说明，程序设定为/bin/cat filename这样的形式，前面的/bin/cat是固定的，用户输入相应的用户名，但是如果我们输入a;new cmd这样，中间用分号隔开，就可以输入新的命令，实现攻击。

同时需要注意，需要加上引号。如果不加上引号，则不会进入/bin/sh。因为没有引号的时候，是两个shell程序，第一个是normal shell，第二个是/bin/sh程序，这种情况下，set-UID不会作用于第二个程序。加上引号以后，则可以变成同样的set-UID程序运行。

这也就是mix code and data带来的问题，解决的机制就是将code和data分开，分别从两个通道进入。例如system函数只有一个参数，可以换成execve函数，它有三个参数，可以将code和data区分开。 并且system使用shell程序作为中间跳板，这会导致非常大的安全隐患，而execve则不通过任何中间跳板，直接调用系统程序。

Capability Leaking

用超人的例子说明，一个普通人穿过超人制服去救人，在救人的过程中他拥有一把可以打开银行的钥匙，在他将制服归还以后，这把钥匙还在他的口袋中，那么就算他的高权限被取消以后，还是会有进入银行的权限。这也就是下面的程序所展示的，首先程序打开/etc/zzz文件夹，这个文件夹只能由root打开，然后我们将程序的set-UID权限归还，虽然这时有效用户ID变成了原用户，但是文件夹并没有关闭，继续执行依然可以打开/bin/sh程序。

Attack Surface and Summary

总结来说，set-UID机制是不合理的，因为即使程序只需要一点特权，通过set-UID机制可以给程序所有权限。这也就是为什么安卓等系统没有set-UID机制，手机里面的权限都是一个一个进行permission。

Section 4 Buffer-Overflow Attacks

Memory layout & Stack

Memory layout

如图右边是内存布局，程序的存储分为Stack、Heap、BSS、Data和Text。

  • Stack

    局部变量存储在栈中

  • Heap

    动态分配的空间存储在堆中，ptr指针本身存储在栈中，但是ptr指向的地址是在堆中。

  • BSS segment

    用于全局变量存储，通常存储未初始化的全局变量。例如图中的静态变量y。

  • Data segment

    用于全局变量存储。例如全局变量x。

  • Text segment

    用处字符串的存储。

Buffer overflow vulnerability

Copy Data to Buffer

strcpy()函数中，编译器会以\0为结尾，因此如果拷贝的字符串自身带\0，那么就会出现拷贝不完整的问题。同时，该函数不检查字符串长度，因此会存在内存溢出的问题。 内存溢出会导致三种不同情况：

  1. 重写入的地址无效，系统杀死进程，程序崩溃。
  2. 重写入的地址有效，系统跳转执行。
  3. 重写入的地址有效，但是无权限，例如跳转到内核，系统不允许，程序崩溃。
